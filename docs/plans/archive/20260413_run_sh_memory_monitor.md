# 구현 계획: run.sh 실시간 메모리 감시 상단 바 추가

## 1. 개요
사용자가 `run.sh`를 통해 서버를 실행 중일 때, 터미널 최상단 한 줄을 **고정 상태 창**으로 활용하여 실시간 시스템 메모리(RAM/VRAM) 사용량을 표시합니다. 서버 로그는 이 상태줄 아래에서만 스크롤되도록 설정하여 가독성을 극대화합니다.

## 2. 목표
- **실시간 모니터링**: 서버 구동 중 약 2초 간격으로 메모리 상태 업데이트.
- **레이아웃 고정**: ANSI Escape Code(`\e[2;r`)를 사용하여 1행을 고정하고 2행부터 로그가 흐르도록 설정.
- **Apple Silicon 최적화**: macOS의 `memory_pressure` 및 `vm_stat` 데이터를 활용하여 유의미한 정보(Used/Free/Pressure) 표시.
- **안전한 종료**: 서버 종료 시 백그라운드 모니터를 중단하고 터미널의 스크롤 영역을 원래대로 복구.

## 3. 상세 설계

### 3.1. 기술적 메커니즘
1. **Scrolling Region (`\e[2;r`)**: 터미널의 스크롤 범위를 2행부터 끝까지로 제한합니다. 1행은 스크롤의 영향을 받지 않는 'Frozen Zone'이 됩니다.
2. **Background Process**: `&`를 이용해 상태를 수집하고 `tput cup 0 0`으로 1행에만 정보를 출력하는 함수를 실행합니다.
3. **Trap**: 서버가 중단(Ctrl+C)되거나 종료될 때 `printf '\e[r'`를 호출해 스크롤 범위를 전체로 복원하고 백그라운드 PID를 `kill` 합니다.

### 3.2. 상태 표시줄 구성 (예시)
```text
[MLX-STATS] RAM: 64.2GB / 128.0GB (50%) | Pressure: 15% | [Model: qwen2.5-7b]
----------------------------------------------------------------------
(서버 로그 시작...)
INFO:     Started server process [1234]
...
```

### 3.3. 주요 함수 추가/수정
- `monitor_memory()`: (신규) 백그라운드 루프. `vm_stat` 등을 파싱하여 최상단에 출력.
- `start_server()`: (수정)
    - 실행 직전 `monitor_memory &` 실행.
    - `trap` 추가 (스크롤 복구 및 모니터 종료).
    - 스크롤 제한 설정.

## 4. 작업 단계
1. **메모리 파싱 로직 검토**: macOS에서 경량으로 메모리 정보를 가져올 수 있는 명령 (`vm_stat`, `top`) 확인.
2. **run.sh 수정**: `monitor_memory` 함수 구현 및 `start_server` 내 제어 로직 삽입.
3. **UI 폴리싱**: ANSI 색상을 적용하여 상태줄을 시각적으로 구분.

## 5. 검증 계획

- [ ] 서버 로그가 출력되어도 상태줄이 밀려나지 않고 제자리를 지키는가?
- [ ] 서버 종료(Ctrl+C) 시 상태줄이 사라지고 터미널 환경이 정상 복구되는가?
- [ ] 여러 번 재기동했을 때 백그라운드 프로세스가 중복으로 쌓이지 않는가?
