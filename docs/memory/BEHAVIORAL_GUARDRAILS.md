## 🚦 TOP 5 핵심 행동 수칙 (High Priority)

> [!IMPORTANT]
> 에이전트는 매 세션 시작 시 이 섹션을 최우선으로 내재화해야 합니다.

1.  **[SDD] 다중 Task 실행 전 설계 문서화 강제 (Project Root 우선)**
    *   **내용**: 단순 파일 수정을 넘어 여러 Task를 수행해야 하는 경우, 반드시 **프로젝트 루트의 `docs/plans/` 폴더**에 계획 문서를 먼저 작성하고 사용자의 승인을 득한다. **Antigravity 전용 아티팩트 경로(숨겨진 로컬 폴더) 사용은 엄격히 금지**한다.
    *   **이유**: 작업의 원자성을 확보함과 동시에, 다른 협업자(사람/타 에이전트)가 작업 내용을 프로젝트 내부에서 즉시 참조할 수 있도록 지식 SSOT를 통합 관리하기 위함.

2.  **[Docs-First] 코드 수정이나 결과 보고보다 명세 업데이트, 자산화 우선**
    *   **내용**: 기능 변경이나 규칙 수정 시 `docs/specs/` 또는 관련 SSOT 문서를 먼저 수정한다. "구현 후 문서화"가 아닌 "문서화 후 구현"을 철저히 지킨다.
    *   **이유**: 코드와 명세의 동기화 오류를 원천 차단하기 위함.

3.  **[Verification] 작업 후 ./verify.sh 실행 및 결과 보고 필수**
    *   **내용**: 모든 코드 변경 후에는 반드시 `./verify.sh`를 실행하고, 그 결과(exitCode, 실패 단계 등)를 보고서 형식으로 제출한다. **Verification 완료 후 `docs/memory/MEMORY.md`를 반드시 최신 상태로 묶어 작성**한다.
    *   **이유**: 회귀 오류를 방지하고 시스템의 안정성을 실시간으로 증명하기 위함.

4.  **[SSOT First] 문서 SSOT 경계를 명확히 구분하고, 변경은 반드시 SSOT에 반영**
    *   **내용**: `docs/specs/`(요구사항), `docs/CRITICAL_LOGIC.md`(결정), `docs/memory/`(세션 지식), `PROJECT_RULES.md`(규칙) 등 **문서 SSOT의 역할을 명확히 구분**하고, 작업 결과는 반드시 **해당 SSOT 문서에 반영**한다. 중복 서술은 금지하며, SSOT 외 문서는 링크만 유지한다.
    *   **이유**: 문서 간 불일치로 인한 혼란을 방지하고, 프로젝트 지식의 일관성과 신뢰성을 유지하기 위함.

5.  **[Latest Doc Protocol] 신규 기능/라이브러리 사용 전, 프로젝트 명세 + 최신 공식 문서 교차 확인**
    *   **내용**: 신규 기능/라이브러리 사용 전, **(1) 프로젝트 내부 `docs/specs/` 명세** 및 **(2) 최신 공식 문서**를 교차 확인하여 프로젝트 정책과 최신 API 표준을 동시 준수한다. 필요 시 웹 검색(`mcp--ddg-search--search`)을 수행한다.
    *   **이유**: 기술 스택의 최신 변화와 프로젝트 정책 간 불일치로 인한 회귀 오류를 방지하기 위함.
